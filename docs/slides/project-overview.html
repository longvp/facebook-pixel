cho Shopify

App nhúng (embedded) giúp merchant quản lý Facebook Pixel & Conversions API (CAPI): CRUD pixel, bật CAPI + access token, theo dõi sự kiện cả phía trình duyệt lẫn server. Toàn bộ được xây dựng cùng Claude Code theo quy trình kỹ thuật.

Facebook Pixel là gì & để làm gì?

Pixel là một đoạn mã theo dõi gắn vào website/storefront. Nó ghi nhận hành vi của khách — xem trang, thêm vào giỏ, mua hàng — rồi gửi các sự kiện đó về Facebook. Nhờ dữ liệu này, merchant chạy quảng cáo hiệu quả hơn:

  • Tối ưu quảng cáo — Facebook học ai có khả năng mua để phân phối ads đúng người.
  • Đo lường chuyển đổi — biết chính xác quảng cáo nào ra đơn, tính ROAS.
  • Retargeting — quảng cáo lại người đã xem sản phẩm / bỏ giỏ nhưng chưa mua.
  • Tệp đối tượng tương tự (lookalike) — tìm khách mới giống khách đã mua.

CAPI (Conversions API) gửi sự kiện từ server song song với pixel trình duyệt, giúp dữ liệu không bị mất khi trình duyệt chặn cookie / ad-blocker → đo lường & tối ưu ads chính xác hơn.

  Phần 3 · Cách Claude làm việc & gỡ lỗi

Subagent-Driven Development

Mỗi task → một subagent mới (context sạch) viết code + test + commit, rồi một subagent reviewer soát spec & chất lượng. Chọn model theo độ khó (rẻ cho việc cơ học, mạnh cho review).

  • Implementer tươi mỗi task → không nhiễu context
  • Review 2 lớp: đúng spec + chất lượng code
  • Ledger tiến độ để không làm lại task đã xong

Systematic Debugging

Luật sắt: tìm root cause trước, không vá triệu chứng. Thu thập bằng chứng ở từng lớp rồi mới sửa. Ví dụ thực tế trong dự án:

  • Token báo "Invalid parameter" → gọi thẳng Graph API, phát hiện user_data rỗng (không phải token sai) → thêm UA+IP. ✓ token vốn hợp lệ
  • Web pixel lỗi RestrictedUrlError (sandbox chặn same-origin) → đổi sang POST cross-origin tới app. ✓ CAPI hoạt động

  Phần 3 · Áp dụng vào việc thật

Muốn làm tính năng mới hoặc fix bug thì flow thế nào?

Tính năng mới — ví dụ: cột “Test event code” + nút Setup

/brainstorm → chốt spec: cột riêng trên Pixel List + nút Setup mở modal đặt test code
/plan → chia task: field DB · action route · UI cột + modal · test
/userstories · /classify → tiêu chí chấp nhận từng story, phân loại now / complex
/execute (TDD) → viết test cột + modal cho đỏ → code Polaris cho xanh → commit
/verify → E2E Playwright thao tác thật trên UI → xanh

Fix bug — ví dụ: token báo “Invalid parameter”

skill systematic-debugging → luật sắt: tìm root cause, không vá triệu chứng
bằng chứng → gọi thẳng Graph API với cùng payload để khoanh vùng lớp lỗi
root cause → user_data rỗng (thiếu UA + IP) — token vốn hợp lệ
test trước → viết test tái hiện lỗi cho đỏ → fix: thêm client_user_agent + IP → xanh
/verify → gửi thật lên Meta → events_received: 1 ✓

Khác biệt chính: tính năng mới đi đủ pipeline brainstorm → verify; fix bug vào thẳng systematic-debugging (bằng chứng → root cause → test tái hiện → fix) — nhưng cả hai đều test trước, có vết, được review qua vòng lặp subagent ở slide trước.

  Phần 5 · Bài học rút ra

Rút ra gì sau khi xây dự án cùng Claude?

TDD giữ AI khỏi “tự tin sai”

Viết test trước buộc Claude chứng minh code chạy thật, không chỉ “trông có vẻ đúng”. Red → Green là lưới an toàn cho code do AI sinh ra.

Bằng chứng > phỏng đoán

Systematic-debugging tìm root cause thật: token “Invalid parameter” hoá ra do user_data rỗng; RestrictedUrlError do sandbox; SaveBar kẹt do App Bridge chặn điều hướng — không vá triệu chứng.

Lưu vết & review từng bước

Spec · plan · flow ghi ra docs/, commit có ngữ cảnh, review 2 lớp. Con người quyết định giữa các bước — AI tăng tốc, không thay quyền duyệt.

Kiểm soát quyền của AI

Hooks + permissions là cần thiết: chặn AI git commit/push, deny đọc .env & lệnh phá hoại. Tự do của agent phải có rào.

Hiểu giới hạn công cụ

Không phải mọi thứ test tự động được — đường App Bridge (SaveBar) không chạm tới ở E2E nên cần verify thủ công. Biết chỗ AI “mù” để bù bằng người.

Re-read context, chốt rõ ý

Đọc lại CLAUDE.md + spec trước mỗi feature giúp Claude làm theo thiết kế hiện tại, không theo trí nhớ. Yêu cầu càng rõ, kết quả càng đúng.
