  Phần 2 · Chức năng app

App làm được gì

CRUD Pixel

List có tìm kiếm, thêm/sửa/xoá; Pixel ID cố định sau khi tạo; nút Edit/Delete + modal xác nhận.

CAPI + Token

Bật CAPI, nhập access token. Khi lưu, app validate token bằng cách gửi 1 event test lên Meta — sai là chặn lưu.

Test event code

Cột riêng + nút Setup → modal đặt test code (route event vào Test Events của Meta).

Web Pixel (client)

Bắn 5 sự kiện: PageView, ViewContent, AddToCart, InitiateCheckout, Purchase — cho mọi pixel active.

Server CAPI

Gửi event song song qua Conversions API (hash email/phone, fbp/fbc, dedup theo event_id).

Home dashboard

Bảng đếm sự kiện theo từng pixel × loại event, cột Is CAPI, nút Refresh.

Shopify SaveBar

Form tạo/sửa dùng contextual SaveBar của App Bridge (Save/Discard ở đỉnh khi có thay đổi).

Audit log

Mỗi event CAPI được ghi lại (trạng thái + phản hồi) phục vụ kiểm tra & thống kê.

Bảo mật & auth

Session-based, validate shop mọi request; mã hoá token (AES-256-GCM) tuỳ cấu hình.

  Phần 3 · Cách Claude làm việc & gỡ lỗi

Subagent-Driven Development

Mỗi task → một subagent mới (context sạch) viết code + test + commit, rồi một subagent reviewer soát spec & chất lượng. Chọn model theo độ khó (rẻ cho việc cơ học, mạnh cho review).

  • Implementer tươi mỗi task → không nhiễu context
  • Review 2 lớp: đúng spec + chất lượng code
  • Ledger tiến độ để không làm lại task đã xong

Systematic Debugging

Luật sắt: tìm root cause trước, không vá triệu chứng. Thu thập bằng chứng ở từng lớp rồi mới sửa. Ví dụ thực tế trong dự án:

  • Token báo "Invalid parameter" → gọi thẳng Graph API, phát hiện user_data rỗng (không phải token sai) → thêm UA+IP. ✓ token vốn hợp lệ
  • Web pixel lỗi RestrictedUrlError (sandbox chặn same-origin) → đổi sang POST cross-origin tới app. ✓ CAPI hoạt động
